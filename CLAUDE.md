# hoshiorange-official - 自己紹介サイト

## プロジェクト概要
hoshiorange の公式ハブサイト。各 SNS（X / YouTube 等）・ブログ・お仕事連絡先など、自分の活動拠点へのリンクを集約する「中継地点」として機能させる。SNS から興味を持って訪れた人が、他の活動にも回遊できることが目的。

| 項目 | 内容 |
|------|------|
| 種別 | 個人公式サイト / リンクハブ |
| 技術スタック | Next.js + TypeScript + CSS Modules（フロントエンドのみ、Tailwind は使わない） |
| バックエンド/DB | 当面なし（必要になったら検討） |
| デプロイ | Vercel |
| ドメイン | 未取得（取得予定） |
| リポジトリ構成 | 単一リポジトリ |

「ぱっと作ってぱっと公開」を優先。追加・改修しやすい構造にしておく。

## Git ワークフロー
- `main` への直接コミット・プッシュは原則禁止
- feature ブランチで作業し、PR 経由で `main` にマージ
- ブランチ命名規則: `feature/{タスク名}` （短く分かりやすく）
- マージはユーザーが承認して行う
- PR が main にマージされたら、ローカルを `main` に切り替えて `git pull` で最新化し、マージ済みの feature ブランチはローカル・リモートとも削除する。次の作業は最新の main から新しい feature ブランチを切って始める。

## ドキュメント管理ルール
- 作業のたびに `docs/NNN_作業名.md` を作成する（NNN は 3 桁の連番）
- ドキュメントには「実施プラン（チェックボックス）」と「完了報告」を含める
- 作業中は逐一チェックボックスを更新し、完了後に完了報告を書く

## PR 作成時の注意
- MCP ツール (`mcp__github__create_pull_request`) の body パラメータに `\n` エスケープシーケンスを使わないこと
- 実際の改行文字を使うか、`gh pr create` コマンドで HEREDOC を使う

## チーム構成

Teams 機能を使う場合、以下の4名体制で構成する。

### リーダー（メインセッション = このClaude Code自身）
- ユーザーの指示を受け取り、適切なチームメートに作業を振り分ける
- 各担当の作業結果をとりまとめてユーザーに報告する
- チームメート間の連携・整合性を管理する
- 詳細は `.claude/rules/leader-rules.md` を参照

### 実装担当
- 作業領域: コーディング全般（FE/BE 分けず一手に引き受ける）
- 作業メモ・仕様理解は `.claude/rules/implementation-notes.md` に記録・更新すること
- ページ追加・コンポーネント実装・スタイリング・ビルド設定など

### コンテンツ・デザイン担当
- 作業領域: 自己紹介の文面・デザイン方針・画像選定・配色など
- 作業メモは `.claude/rules/content-notes.md` に記録・更新すること
- 実装担当に渡すコンテンツ・デザイン指示の整理も担当

### 雑用担当
- 作業領域: 上記担当に明確に当てはまらないその他業務
- 作業メモは `.claude/rules/odd-jobs-notes.md` に記録・更新すること
- 例: git 操作の代行、環境確認、ファイル整理、ツール設定、調査など

## 共通ルール
- 各担当は自分の担当領域の状況をしっかり把握し、記憶ファイルに記録すること
- 記憶ファイルは作業のたびに最新状態に更新すること
- 他の担当の記憶ファイルも参照して、横断的な整合性を保つこと
- 日本語で応答すること

## 環境情報

- Node.js: Next.js 対応の LTS バージョン
